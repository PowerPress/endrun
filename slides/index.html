The only firm requirement for Endrun is a USB port; as we'll see later, that's more powerful than you might think.

Modifiable On the Fly

This isn't an Exchange Server. You know your needs better than we do.

Brendan

We come from the amateur radio community, where any components can be the right components and everything should be interchangeable. With that in mind, Endrun is built using standard, well-tested, open source components and put together using Python, rather than using all custom code. This lets Endrun benefit from decades of work from thousands of programmers (with all the attendant work on security that implies). It also means that Endrun is customizable by programmers in the field; in an emergency situation, this can give flexibility that fully-custom solutions cannot.

Previous Work

Tor

The choice of dissidents and activists everywhere.

Brendan

You've heard of Tor, of course. Tor provides privacy by encrypting communication at every relay using different keys.

Here's the issue: the original Tor paper stated that any adversary who could observe either the entry and exit nodes, or who can control a simple majority of all Tor nodes, can compromise the privacy guarantees. Just using Amazon EC2, it would only take about US$50K/month to double the amount of Tor nodes (thus controlling 50%), so that's a realistic threat from *any* large organization.

With respect to entry/exit monitoring, a Harvard undergraduate who wanted to skip a business ethics final found this out the great way.

AirChat, by Anonymous

Internet-free, end-to-end encrypted text and low-bandwidth voice communication.

By Anonymous: the people who brought you Sabu.

Grant

  • Back in April, Anonymous announced Airchat, a system for secure internet-free communication using amateur radio.
  • Using encryption over amateur radio is illegal in the US according to FCC rules, part 97
  • Anonymous doesn't care; said "fuck you,NSA" (shrug)
  • However, opsec problem: it's an online protocol (continuous communication)
  • Who knows what trangulation is? Who knows what "unidirectional" means?
  • 2 or more unidirectional radio antennas + signal strength meters + some math == you can find radio transmitters really well
  • Anyone serving in S. Pacific during WWII will tell you that this can ruin your day
  • It has been reported that Syria may have used missiles that track satellite phones to kill journalists (250mW)
  • Handheld ham radios usually broadcast at 5W; this is more than enough to locate you; you'll end up with "connection reset by missle"

Neighborhood mesh networks for the Zombie apocalypse

Brendan

Commotion Wireless creates community mesh networks using special firmware for Linksys WiFi routers; it's designed to share an Internet connection in an underserved area. NASA/ESA Delay-Tolerant Networking

Close, but no appropriations

Grant

  • DTN has been around since the mid 90s when Vint Cerf stated working on the InterPlanetary Internet.
  • It solves most of the problems we described above: it'll handle long delays, transient links, multiple transport paths, etc.
  • Most of the funding/research comes from ESA + NASA (delay-tolerant) and DARPA (disruption-tolerant)
  • it's never been implemented the same way twice; the closest thing to standard (Mars Telecom Orbiter) ran into Congress and died a horrible fiery death

  • All implementations to date are binary protocols, which make them hard to parse (hard to parse == hard to secure)

NaCl

Daniel J. Bernstein, University of Illinois - Chicago
  • Bleeding-edge elliptical-curve cryptography

  • Fast and efficient
  • Unfortunately, not guaranteed to run anywhere but DJB's office

Brendan

Git solves our data control problems. However, outside of hashing and the occasional tag or commit signing, it doesn't really do crypto.

We wanted the best crypto we could find. Our intended use cases deserve no less than that. So we looked around, and decided on Salt. Salt is an elliptical-curve cryptography framework developed by Daniel J. Bernstein of the University of Chicago. It is incredibly fast, has very low overhead, and very secure -- OpenDNS uses it to secure their backhaul. However, it's written in C. With hardware-specific compiler flags. That means unless you have a thing for makefiles, it's only guaranteed to run correctly if your name is Daniel J. Bernstein and you're sitting in front of your office computer.

LibSodium

NaCl, as fixed by OpenDNS.

LibSodium is a wrapper around NaCl developed by the friendly folks at OpenDNS.

  • Has bindings in a number of different languages (Python, Ruby, PHP, even Haskell!)
  • Much easier to work with

Brendan

Luckily, this problem has *also* been solved for us. The OpenDNS people wanted to use Salt, and didn't want to sentence their devops team to compiling salt on every server architecture variant they have, so they developed libsodium. Libsodium is a platform- and language-independent wrapper around salt. It has good performance compared to the natively-compiled C binaries, has bindings for most of today's common development languages, and will run on pretty much anything with a PRNG.

We can't directly encrypt the data stored *in* a git repository, so instead we sign and encrypt bundles, then sign them again before they're ready for transport.

Payload Format

Payloads are how Endrun moves data around. Each payload contains:

  • Origin node identifier
  • Destination node identifier
  • Timestamp (TTL)
  • Contents
  • Chain of Custody data
  • Copies

Grant

That brings us to payloads. In Endrun, a payload is a nice little serialized package with the encrypted git bundle and its routing data. We have origin and destination, for which we use the salt public key fingerprints. We have an epoc timestamp that is used for determining time to live, our bundle, and then we have our chain of custody data -- more about that in a minute.

Provisioning the cell

When creating an Endrun cell of 8 nodes:

  • A "source" repo is created with 8 branches
  • 8 double keypairs are created
  • 8 bundles are created, 1 for each branch

Grant

  • Provisioning an endrun cell works like this.
  • the source repo is created because we need a common starting point
  • When provisioning is complete, we destroy this repo
  • Double keypairs are encryption keys and signing keys

Provisioning the nodes

Then, on each node:

  • A repo is created with 7 remotes (everyone except itself) and 8 branches
  • The public part of 7 double keypairs are copied over
  • Node gets its own full double keypair
  • SQLite databases are set up
  • Endrun daemon is started

Grant

  • Provisioning cells is sort of a round robin model
  • If I have N nodes, each node gets N-1 bundles and N-1 public keys

Moving data

When using the node, all commits occur on the master branch.

When we move data:

  • Master gets merged into our destination's branch
  • Branch is pushed to the destination's remote (which is a bundle)
  • Payload is created and moved to the destination node
  • Destination does a pull to its named branch
  • Branch is merged into master

Grant

Routing

Grant

Now we talk about routing.

Pairwise Communication

DTN routing occurs between pairs of nodes.

Grant

  • Node connection is pairwise
  • All data needed to complete the transmission is contained within the payload
  • Nodes never need to be contacted for more information
  • Think about passing notes in class: you pass it by looking for the next closest person, destination is written on the note
  • We can also wait until the teacher isn't looking, then pass: this is store and forward

Spray and Pray Wait

  • Replication-based routing uses too much bandwidth.
  • Forwarding-based routing don't handle transient links well.
  • Spray and Wait combines forwarding-based routing with a limit on message copies to prevent flooding.

Grant

Multiple ways to pass notes: one copy to one person at a time(forwarding-based routing), or multiple copies to everyone (replication-based or epidemic routing)

  • One copy might not get there
  • Multiple copies to everyone clogs up the network

Best approach is spray and wait

  • Developed at university of southern california
  • Combines the epidemic routing benefits with a limit on copies to prevent congestion

Grant

Basic idea

  • Set an upper bound, in this case, 8.
  • First round, the payload is sent to 8 nodes (or every node we can find).
  • Second round, every node with a copy sends it to 4 nodes (half rounded down)
  • Third round, every node with a copy sends it to 2 nodes (half rounded down)
  • Fourth round, nodes should only have one copy; they hold onto it and wait for direct contact (or TTL death)

Grant

Chain of Custody

Endrun does not maintain a routing table, or any global knowledge of the network. This has some downsides:

  • Nodes can be compromised without immediate knowledge
  • Bandwidth can be wasted trying to contact nodes that have moved or are no longer available
  • I can't route data to people I don't know

Grant

Chain of custody solves these issues without requiring global knowledge

We set up chain of custody in 3 parts: a record on the payload itself, a receipt table, and a map table

The Chain of custody on the node is just an array of receipts.

Each receipt is recipient node ID, timestamp of receipt, and hash of the payload at that time.

Everytime the payload is passed, a receipt is appended to the array.

Receipt Table

Grant

Receipt table lives on the node's SQLite DB

Same receipt, but only for that node

Map Table

Grant

Map table is also in SQLite DB

Every time one node encounters another (by receiving data from it) a record is made in the table.

Anytime we prepare to hand off data, we check this table first; if we haven't "seen" a node in a while, we avoid it
