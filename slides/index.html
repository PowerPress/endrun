We've made a system called Endrun for securely communicating without the internet. Here's what we want you to take away from our presentation.

No Internet Use

We're making an end run around the Internet. Because someone has to.

Brendan

Endrun is designed to make an ``end run'' around the Internet. Reasons:

  • internet failure due to natural disaster, accident, hostility
  • internet is untrustworthy (Warning: don't ramble)
  • internet connectivity not feasible due to remoteness or terrain

By Any Means Necessary

  • Sneakernet
  • XBee
  • Sneakernet+
  • Carrier Pigeon
  • Amateur Radio*

Brendan

Endrun is "by any means necessary." Endrun is designed to work across any form of communication that is available, whether that's physical data movement and low-power transient unlicensed radio links, as we would recommend, or by Earth-Moon-Earth (EME) high-powered signals from fix points, or even amateur (ham) radio. The only firm requirement for Endrun is a USB port; as we'll see later, that's more powerful than you might think.

Old-School Cellular Communications

Cell as in "group", not cell as in "for the enjoyment of all patrons, please turn off your"

Brendan

Endrun is designed for communications between small groups of people---``cells'' or ``pods.'' It is not designed for open-ended communication between people that have no prior relationship.

Disruption/Delay-Tolerant

  • Conflict zones
  • Disaster areas
  • Hostile terrain (inc. N. Scotland, Scandinavia, the southern U.S.)
  • SPAAAAAAAAAACE!

Grant

Because our system can be used in really nasty places where the trains don't always run on time, Endrun is disruption- and delay-tolerant.

It is designed to work where nodes have no real- or bounded-time connection to each other

Because it's medium-agnostic, will work in places where the internet has traditionally been difficult to use

Modifiable On the Fly

This isn't an Exchange Server. You know your needs better than we do.

Brendan

We come from the amateur radio community, where any components can be the right components and everything should be interchangeable. With that in mind, Endrun is built using standard, well-tested, open source components and put together using Python, rather than using all custom code. This lets Endrun benefit from decades of work from thousands of programmers (with all the attendant work on security that implies). It also means that Endrun is customizable by programmers in the field; in an emergency situation, this can give flexibility that fully-custom solutions cannot.

Previous Work

Tor

The choice of dissidents and activists everywhere.

Brendan

You've heard of Tor, of course. Tor provides privacy by encrypting communication at every relay using different keys. However, exit nodes are the weakpoint, and an adversary with sufficient resources could control the majority of end nodes.

AirChat, by Anonymous

Internet-free, end-to-end encrypted text and low-bandwidth voice communication

Grant

Anonymous came up with a Airchat provides secure encrypted messaging, but uses amateur radio and is illegal in the US under part 97 of the FCC rules. Other countries have similar restrictions, so it's illegal there too. It also relies on semi-permanent high-power radio links, which means anyone with a couple of directional antennas can ruin your day by calling in air support, or as we call it, "connection reset by missile".

Commotion Wireless / Project Byzantium

Neighborhood mesh networks for the Zombie apocalypse

Brendan

Commotion Wireless creates ad-hoc mesh networks using regular 802.11a/b/g/n hardware. Pretty cool. In a "zombie apocalypse", this would work pretty well. However, wifi is semi-permanent wireless radio. Wifi has the same problems we described above.

NASA/ESA Delay-Tolerant Networking

Close, but no appropriations

Grant

DTN has been around since the early 90s when Vint Cerf stated working on the InterPlaNet. It solves most of the problems we described above: it'll handle long delays, transient links, multiple transport paths, etc. However, security is a second-tier concern if it's considered at all. Also, for the most part, DTN has been an academic exercise, limited to university research teams with grant funding from NASA or ESA. It's never been implemented the same way twice, and the closest we've come to having a standard implementation of it -- in the form of NASA's proposed Mars Telecommunications Orbiter -- fell victim to the Appropriations Committee of the United States Congress.

Endrun 101

Grant

  • DTN is the closest existing concept to what we want to do.
  • However, it's never been implemented the same way twice
  • Binary protocol, which makes it hard to parse
  • Security is an afterthought at best
  • So we built our own.

Git

Git solves many of the problems that a DTN faces.

  • Information order
  • Information integrity
  • Information divergence

Grant

There are a lot of problems with putting together a networking protocol

  • stuff shows up in the wrong order, stuff shows up mangled
  • stuff... doesn't show up, stuff shows up twice.
  • Changes to the same information need to be incorporated back into the whole
  • Same problems occur in distributed software teams
  • Git solves the problem. Beautifully.

Pretty nifty, right? However, you're probably thinking "Don't you need a network connection for Git?"

Git Bundles

Bundles let us "fake" remote repos.

From the man page:

	    This command provides support for git fetch and git pull 
	    to operate by packaging objects and references in an 
	    archive at the originating machine, then importing those 
	    into another repository using git fetch and git pull after 
	    moving the archive by some means.

Look, ma! No network connection!

Grant

Git has a feature called "bundles" that are a way of using a file to connect to a remote repo instead of a network connection.

Originally intended for air-gap situations

  • Create bundle at point A, push to it
  • Copy file to USB, burn to CD, print in 6-point font with 1cm margins and give to US Attorney
  • Copy file onto machine at point B, tell git where to find it in config
  • Git pull from bundle

NaCl

NaCl is a cryptography suite that was developed by Daniel J. Bernstein of the University of Chicago.

  • Bleeding-edge elliptical-curve cryptography

  • Fast and efficient
  • Unfortunately, not guaranteed to run anywhere but DJB's office

Brendan

Git solves our data control problems. However, outside of hashing and the occasional tag or commit signing, it doesn't really do crypto.

We wanted the best crypto we could find. Our intended use cases deserve no less than that. So we looked around, and decided on Salt. Salt is an elliptical-curve cryptography framework developed by Daniel J. Bernstein of the University of Chicago. It is incredibly fast, has very low overhead, and very secure -- OpenDNS uses it to secure their backhaul. However, it's written in C. With hardware-specific compiler flags. That means unless you have a thing for makefiles, it's only guaranteed to run correctly if your name is Daniel J. Bernstein and you're sitting in front of your office computer.

LibSodium

LibSodium is a wrapper around NaCl developed by the friendly folks at OpenDNS.

  • Has bindings in a number of different languages (Python, Ruby, PHP, etc.)
  • Much easier to work with

Endrun uses LibSodium to sign, encrypt, and sign bundles before they are added to Endrun payloads.

Brendan

Luckily, this problem has *also* been solved for us. The OpenDNS people wanted to use Salt, and didn't want to sentence their devops team to compiling salt on every server architecture variant they have, so they developed libsodium. Libsodium is a platform- and language-independent wrapper around salt. It has good performance compared to the natively-compiled C binaries, has bindings for most of today's common development languages, and will run on pretty much anything with a PRNG.

We can't directly encrypt the data stored *in* a git repository, so instead we sign and encrypt bundles, then sign them again before they're ready for transport.

Payload Format

Payloads are how Endrun moves data around. Each payload contains:

  • Origin node identifier
  • Destination node identifier
  • Timestamp (TTL)
  • Contents
  • Chain of Custody data
  • Copies

Grant

That brings us to payloads. In Endrun, a payload is a nice little serialized package with the encrypted git bundle and its routing data. We have origin and destination, for which we use the salt public key fingerprints. We have an epoc timestamp that is used for determining time to live, our bundle, and then we have our chain of custody data -- more about that in a minute.

Provisioning the cell

When creating an Endrun cell of 8 nodes:

  • A "source" repo is created with 8 branches
  • 8 double keypairs are created
  • 8 bundles are created, 1 for each branch

Grant

  • Provisioning an endrun cell works like this.
  • the source repo is created because we need a common starting point
  • When provisioning is complete, we destroy this repo
  • Double keypairs are encryption keys and signing keys

Provisioning the nodes

Then, on each node:

  • A repo is created with 7 remotes (everyone except itself) and 8 branches
  • The public part of 7 double keypairs are copied over
  • Node gets its own full double keypair
  • SQLite databases are set up
  • Endrun daemon is started

Grant

  • Provisioning cells is sort of a round robin model
  • If I have N nodes, each node gets N-1 bundles and N-1 public keys

Moving data

When using the node, all commits occur on the master branch.

When we move data:

  • Master gets merged into our destination's branch
  • Branch is pushed to the destination's remote (which is a bundle)
  • Payload is created and moved to the destination node
  • Destination does a pull to its named branch
  • Branch is merged into master

Grant

Routing

Grant

Now we talk about routing.

Pairwise Communication

DTN routing occurs between pairs of nodes.

Grant

  • Node connection is pairwise
  • All data needed to complete the transmission is contained within the payload
  • Nodes never need to be contacted for more information
  • Think about passing notes in class: you pass it by looking for the next closest person, destination is written on the note
  • We can also wait until the teacher isn't looking, then pass: this is store and forward

Spray and Pray Wait

  • Replication-based routing uses too much bandwidth.
  • Forwarding-based routing don't handle transient links well.
  • Spray and Wait combines forwarding-based routing with a limit on message copies to prevent flooding.

Grant

Multiple ways to pass notes: one copy to one person at a time(forwarding-based routing), or multiple copies to everyone (replication-based or epidemic routing)

  • One copy might not get there
  • Multiple copies to everyone clogs up the network

Best approach is spray and wait

  • Developed at university of southern california
  • Combines the epidemic routing benefits with a limit on copies to prevent congestion

Spray and Wait

Grant

Basic idea

  • Set an upper bound, in this case, 8.
  • First round, the payload is sent to 8 nodes (or every node we can find).
  • Second round, every node with a copy sends it to 4 nodes (half rounded down)
  • Third round, every node with a copy sends it to 2 nodes (half rounded down)
  • Fourth round, nodes should only have one copy; they hold onto it and wait for direct contact (or TTL death)
